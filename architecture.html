Cloudflare Infrastructure Blueprint

Infrastructure Only

This page shows the Cloudflare stack for a first-stage prototype: UI, backend, storage, async jobs, auth, and observability. Business workflow is intentionally removed.
Primary Stack
Pages, Workers, R2, D1, KV, Queues, Turnstile / Access, Logs.
Design Rule
Keep the architecture Cloudflare-only until the prototype proves a clear need for more infrastructure.

三、Recommended Cloudflare Components

Pages

Use Pages for the UI. It is the simplest way to host a prototype frontend with fast global delivery.

UI Static Hosting

Workers / Functions

Use Workers for API routes, file processing, auth checks, and orchestration between storage and async jobs.

API Orchestration

R2 + D1 + KV

Use R2 for files, D1 for relational data, and KV for fast session or draft state. This is a clean Cloudflare-native data split.

Files Records Session State

Queues + Cron

Use Queues for async work and Cron for scheduled tasks such as cleanup, sync, or periodic status checks.

Async Scheduled Tasks

四、Optional Add-ons

Turnstile / Access: add when you need anti-bot protection or private access control.
External AI API: keep optional for extraction or summarization, not part of the core Cloudflare stack.
Analytics / Logs: use for debugging, usage insight, and prototype validation.
